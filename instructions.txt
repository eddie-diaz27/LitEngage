## PROJECT OVERVIEW

You are building a complete AI-powered book recommendation system for a school library to increase student reading engagement. The system mimics a helpful librarian who provides personalized book recommendations based on student reading history, preferences, and reviews.

**Core Value Proposition:** Librarians observe that students read books recommended by them, based on what similar students enjoyed. This system scales that personalized recommendation capability using LLMs and RAG.

**Target Users:**
- Primary: 6-12 students (ages 11-18) seeking book recommendations
- Secondary: Librarians monitoring recommendations and managing the catalog

**Key Constraints:**
- MUST comply with COPPA (Children's Online Privacy Protection Act) - no student PII to external APIs
- MUST comply with FERPA (Family Educational Rights and Privacy Act) - protect educational records
- MUST provide age-appropriate, reading-level-appropriate recommendations only
- MUST have safety guardrails to prevent harmful content

### 2. DEEPTEAM GUARDRAILS INTEGRATION

**Implementation Requirements:**
- Create `backend/services/guardrails.py` with all guards configured
- Guards MUST be initialized ONCE at app startup (not per request - expensive!)
- MUST use async methods (`a_guard_input`, `a_guard_output`) for better performance

**Input Guardrail Flow:**
```python
async def check_input_guardrails(message: str) -> tuple[bool, str]:
    """Returns (is_safe, fallback_message_if_unsafe)"""
    result = await guardrails.a_guard_input(
        llm_input=message,
        expected_output="A response about book recommendations"
    )
    
    if result.safe:
        return True, ""
    
    # Determine specific breach type
    if result.prompt_injection:
        return False, "I can only help with book recommendations and library questions!"
    elif result.topical_breach:
        return False, "Let's keep our conversation focused on books and reading. How can I help you find a great book?"
    else:
        return False, "I'm having trouble understanding that request. Could you rephrase it?"
```

**Output Guardrail Flow (with retries):**
```python
MAX_RETRIES = 3

async def check_output_guardrails(response: str, context: str, attempt: int = 1) -> tuple[bool, str]:
    """Returns (is_safe, safe_response_or_original)"""
    result = await guardrails.a_guard_output(
        llm_output=response,
        llm_input=context
    )
    
    if result.safe:
        return True, response
    
    # Log the breach
    logger.warning(f"Output guardrail breach on attempt {attempt}: {result.breaches}")
    
    if attempt >= MAX_RETRIES:
        # Use safe fallback after max retries
        return False, "I apologize, but I'm having trouble generating a good recommendation right now. Would you like to try searching for a different type of book?"
    
    # Trigger retry (handled by caller - regenerate with modified prompt)
    return False, ""
```

### 4. VECTOR STORE (CHROMADB) SETUP

**Embedding Pipeline:**
```python
def create_book_document(book: Book) -> Document:
    """Convert book record to LangChain Document for embedding."""
    
    # Combine semantic fields into rich text
    content = f"{book.title} by {book.author}. "
    
    if book.description:
        content += f"{book.description} "
    
    if book.genres_json:
        genres_str = ", ".join(book.genres_json)
        content += f"Genres: {genres_str}. "
    
    if book.themes_json:
        themes_str = ", ".join(book.themes_json)
        content += f"Themes: {themes_str}."
    
    # Metadata for filtering (NOT embedded)
    metadata = {
        "book_id": book.id,
        "title": book.title,
        "author": book.author,
        "reading_level": book.reading_level,
        "age_appropriate": book.age_appropriate,
        "publication_year": book.publication_year,
        "avg_rating": book.avg_rating or 0,
        "genres": book.genres_json or [],
    }
    
    return Document(page_content=content, metadata=metadata)
```

**Search Implementation (MMR for Diversity):**
```python
def search_books_semantic(
    query: str,
    reading_level: str,
    genres: List[str] = None,
    k: int = 10
) -> List[dict]:
    """Semantic search with metadata filtering and diversity."""
    
    # Build metadata filter
    filter_dict = {"reading_level": reading_level}
    if genres:
        filter_dict["genres"] = {"$in": genres}
    
    # MMR search (70% relevance, 30% diversity)
    results = vectorstore.max_marginal_relevance_search(
        query=query,
        k=k,
        fetch_k=k * 3,  # Fetch 3x, then pick diverse k
        lambda_mult=0.7,
        filter=filter_dict
    )
    
    # Convert to book dicts
    return [
        {
            "book_id": doc.metadata["book_id"],
            "title": doc.metadata["title"],
            "author": doc.metadata["author"],
            "relevance_score": doc.metadata.get("score", 0)
        }
        for doc in results
    ]
```

- Set `sample_rate=1.0` initially, reduce if latency issues
- Log ALL breaches for audit
- Test guardrails thoroughly (try to break them!)
- Have clear fallback messages for each breach type

### LangGraph Agent Best Practices
- Keep state schema minimal (only what you need)
- Use `add_messages` reducer for conversation history
- Bind tools with CLEAR, SPECIFIC descriptions (LLM needs good descriptions to call them)
- Test agent with edge cases (no results, ambiguous queries, etc.)
- Use LangSmith to debug agent behavior
- Implement timeout for tool execution (5-10 seconds max)

### Vector Search Best Practices
- ALWAYS filter by reading level first (hard constraint)
- Use MMR for diversity (lambda=0.7)
- Fetch 3x more results than needed, then filter
- Cache embeddings (don't regenerate unnecessarily)
- Consider adding collaborative filtering later (if you have interaction data)

### Performance Optimization
- Use async/await throughout FastAPI
- Connection pooling for database
- Semantic caching for common queries (Redis)
- Batch embeddings generation
- Lazy load ChromaDB (don't load entire index into memory)

### Error Handling
- Never expose internal errors to students
- Log detailed errors server-side
- Show friendly messages to users
- Implement retry logic for LLM API calls (with exponential backoff)
- Have circuit breakers for external services
